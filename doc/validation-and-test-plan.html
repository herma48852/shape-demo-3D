Validation & Testing Infrastructure

This document defines the isolated stubs, visual checks, and mock infrastructures required to validate each phase of the Interactive Controls implementation before moving to the next.

Phase 1: ImGui Compilation & OpenGL Hook Validation

Goal: Ensure CMake successfully compiled ImGui via FetchContent and that the GLFW/OpenGL backends can render over your 3D viewport without capturing mouse events meant for the camera.

Test Stub (Inject into src/main.cpp loop):

Do not build your custom UI yet. Use the built-in ImGui Demo Window to verify font rendering, interaction, and context setup.

ImGui_ImplOpenGL3_NewFrame();
ImGui_ImplGlfw_NewFrame();
ImGui::NewFrame();

// STUB: Render standard ImGui diagnostic window
bool show_demo = true;
ImGui::ShowDemoWindow(&show_demo);

ImGui::Render();
ImGui_ImplOpenGL3_RenderDrawData(ImGui::GetDrawData());
Acceptance Criteria:
  • The complex ImGui Demo window appears rendered cleanly over the dark blue background.
  • Clicking and dragging the window works smoothly (verifies GLFW input capture).
  • Clicking the 3D space behind the window still rotates the camera (verifies input fallthrough).

Phase 3: Kinematics & Physics Isolation Test

Goal: Ensure the math for linear bouncing and trigonometric orbits works correctly before piping those rapidly changing coordinates into the DDS network.

Rendering Stub:

Pipe the active_publishers kinematics loop directly to the OpenGL graphics.draw_cube() method, completely ignoring the DDS Reader queue.

// INSIDE KINEMATICS LOOP

    // STUB: Render directly to screen based on local physics state (Bypass DDS)
    graphics.draw_cube(pub.x, pub.y, pub.z, pub.size, 1.0f, 1.0f, 1.0f, false);
    
    // dds_manager.write_shape_sample(...); // KEEP COMMENTED OUT
}
Acceptance Criteria:
  • Creating a publisher in the UI immediately spawns a white cube in the 3D viewport.
  • The cube bounces predictably between the 0 and 250 coordinate bounds without flying off-screen.

Phase 4: Full Stack Loopback Integration

Goal: Remove the stubs and validate that the data correctly traverses down into the C++ DDS Publisher, crosses the UDP loopback interface, is caught by the DDS Subscriber, passed to the lock-free queue, and finally rendered as a wireframe.

Final Execution Test:
  • Uncomment dds_manager.create_writer and dds_manager.write_shape_sample.
  • Run the application. The 3D viewport should be empty (local publishers render as data, not direct pixels).
  • Use the ImGui UI to create a BLUE Cube Publisher. (Viewport should still be empty).
  • Use the ImGui UI to create a Subscriber filtering for "BLUE".
  • SUCCESS: A blue wireframe cube should immediately appear in the viewport, following the kinematics trajectory, proving the entire C++ DDS pipeline is healthy.
